== Приложение А: Примеры потоков создания ценности

В этом разделе демонстрируется, как цепочка создания ценности услуг может быть применена к практическим ситуациям, и приведены примеры потоков создания ценности. Эти потоки создания ценности показывают, как может протекать активность через цепочку создания ценности. Это не модели для копирования, а просто примеры для понимания того, как следует использовать цепочку создания ценности.

Примеры включают некоторые примерные рабочие роли. Это просто роли, которые могут существовать в описываемой вымышленной организации, и они не являются рекомендуемыми ролями для каждой организации. Для облегчения понимания первый поток создания ценности описан подробно; в последующих примерах предоставлена только таблица.

=== А.1 Пользователю необходимо разрешить инцидент

В этом первом примере потока создания ценности WiFi на складе работает неправильно из-за выхода из строя точки беспроводного доступа. Это оказывает значительное влияние на бизнес, потому что водитель погрузчика не может получать инструкции достаточно быстро, и, таким образом, существует риск срыва бизнес-срока. Это может показаться относительно простым инцидентом; однако его нельзя разрешить, просто механически следуя шагам заранее определенной процедуры управления инцидентами.

Во-первых, кто-то должен заметить, что есть инцидент, и знать, как его сообщить, и должна быть возможность для этого человека точно передать срочность ситуации, чтобы ее можно было правильно приоритезировать. Человек, получающий сообщение, должен иметь как полномочия для эскалации инцидента, так и процедуры для этого и для мониторинга прогресса инцидента. Должны быть ресурсы, позволяющие достаточно быструю эскалацию; кто-то должен обладать навыками, знаниями и инструментами, необходимыми для расследования инцидента; и должны быть процедуры, позволяющие внедрять стандартные изменения без необходимости получения дополнительного утверждения. Должна быть возможность для кого-то получить доступ к точной конфигурационной информации и зарегистрировать ремонт после его завершения. Также должна быть возможность зарегистрировать, что расходная деталь была использована, и перезаказать ее на будущие нужды. Однако, если ремонт будет иметь какую-либо ценность, склад должен быть проинформирован о том, что произошло, чтобы можно было возобновить нормальную работу. Также важно проверить, насколько хорошо был разрешен инцидент, чтобы увидеть, есть ли какие-либо уроки, которые можно извлечь.

| Получать/создавать
| Разработка и управление программным обеспечением, валидация и тестирование услуг, проектирование услуг
| Команды разработки программного обеспечения
| Каждая команда разработки управляет бэклогом и разрабатывает код для областей, назначенных им. Каждая команда также разрабатывает тесты для включения в автоматизированный конвейер. Весь код автоматически интегрируется и тестируется два раза в день, обеспечивая совместную работу кода, написанного разными командами.

| Получать/создавать, Проектировать и переходить
| Проектирование услуг, управление организационными изменениями, управление развертыванием, управление конфигурацией услуг
| Команды разработки программного обеспечения
| Развертывание нового программного обеспечения запускается, как только оно готово. Для этого не требуются индивидуальные запросы на изменение, так как оценка риска была проведена ранее, а автоматизация обеспечивает развертывание кода точно в соответствии с планом. Конфигурационные данные используются для управления развертыванием, поэтому не требуется отдельной активности для их обновления.
